# 第 3 课作业 · 线程池、共享状态与 Send

## 作业目标

把并发代码从“能跑”推进到“边界清楚、测试确定”。重点是 `Send` 约束、共享状态和无 sleep 的测试。

## 代码位置

| 用途 | 路径 |
|---|---|
| 并发与内存模型 | `projects/v2-concurrency` |

## 需求

1. 增加 `execute_with_result<T: Send + 'static>(f) -> Receiver<T>`。
2. 使用 `Arc<RwLock<HashMap<String, u64>>>` 实现并发词频统计。
3. 有意触发一次 E0277：把 `Rc` 或 `RefCell` 移入任务，记录编译器错误，再用 `Arc` 或 `Arc<Mutex<_>>` 修复。
4. 每个新增能力至少有 2 个确定性测试。
5. 测试不能依赖 `sleep`。

## PR 说明必须包含

- E0277 的原始含义，以及为什么 `Rc` / `RefCell` 不能跨线程；
- 修复方案为什么满足 `Send` 或 `Sync`；
- 词频统计中读写锁的粒度选择。

## 验证命令

```bash
cargo test -p v2-concurrency
cargo clippy -p v2-concurrency --all-targets -- -D warnings
cargo fmt -p v2-concurrency --check
```

## 自检清单

- [ ] `execute_with_result` 能把任务结果传回调用方。
- [ ] 词频统计结果稳定，不依赖线程执行顺序。
- [ ] PR 中的 E0277 说明来自真实编译器输出。
