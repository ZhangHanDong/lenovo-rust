# 第 1 课作业 · 类型状态与非法状态

## 作业目标

把连接生命周期从“文档约定”改成“类型约束”。本课作业关注 typestate、终止状态和运行时容量错误的边界划分。

## 代码位置

| 用途 | 路径 |
|---|---|
| 类型驱动设计 | `projects/v2-type-modeling` |
| 惯用法对照 | `projects/v2-idiomatic` |

## 需求

1. 在 `v2-type-modeling` 中新增 `Closed` 终止状态。
2. 为 `Connection<Authenticated>` 和 `Connection<Connected>` 增加 `close(self) -> Connection<Closed>`。
3. `Closed` 状态不能暴露任何后续状态转换方法。
4. 为 `send` 增加容量限制语义，超过限制时返回 `Result`。
5. 增加至少 2 个 `#[test]`，覆盖正常发送、容量超限或关闭流程。
6. 增加 1 个 `compile_fail` 示例，证明 `Connection<Closed>` 不能发送数据。

## PR 说明必须包含

- 哪些规则被类型系统保证；
- 哪些规则仍然必须用 `Result` 表达；
- 为什么 `Closed` 是终止状态，而不是普通运行时标志。

## 验证命令

```bash
cargo fmt --check
cargo clippy --all-targets -- -D warnings
cargo test -p v2-idiomatic
cargo test -p v2-type-modeling
```

## 自检清单

- [ ] 关闭后的连接没有 `send` 方法。
- [ ] 容量超限不会 panic。
- [ ] 编译失败示例表达的是非法 API 使用，而不是拼写或导入错误。
