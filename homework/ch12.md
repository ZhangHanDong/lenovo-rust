# 第 12 课作业 · Windows GUI 与系统集成

## 作业目标

在桌面 GUI / 服务 / 注册表 / 托盘菜单之间建立清晰边界，让平台代码不破坏跨平台构建。

## 代码位置

| 用途 | 路径 |
|---|---|
| GUI 与系统集成 | `projects/v2-win-gui` |

## 需求

1. 为服务增加 Pause / Continue SCM control，使用 `ServiceControlAccept::PAUSE_CONTINUE` 驱动核心状态机。
   - 提示：`service.rs` 目前的 `event_handler` 只处理 `Interrogate`/`Stop`（见课本 §2.3）；你要新增 `ServiceControl::Pause`/`ServiceControl::Continue` 分支，并像现有 `Stop` 分支一样通过 channel 把意图交给主循环，再由主循环调用 `ServiceState::apply` 驱动迁移、`set_service_status` 汇报。
   - 汇报 `Paused`/`Running` 时，`controls_accepted` 要包含 `ServiceControlAccept::PAUSE_CONTINUE`（`Stopped`/`Running` 之间目前只声明了 `STOP`）。
2. 增加至少 1 个平台无关状态机测试（覆盖 `Running --Pause--> Paused --Continue--> Running` 这条新路径，可仿照 `lib.rs` 里已有的 `service_state_machine_happy_path`）。
3. 在 `integration` 中增加注册表读取：`HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProductName`。
   - 提示：用 `RegOpenKeyExW`（只读，`KEY_READ` 即可，不需要 `KEY_SET_VALUE`）+ `RegQueryValueExW`，读出来的字节按 UTF-16 解码、去掉尾部 NUL——这是课本 §2.4 里 `reg_sz_bytes` 编码方向的逆操作。
4. 对外返回平台中立的 `Option<String>`，非 Windows 平台提供 stub（返回 `None` 即可，函数签名要在两个平台上一致，调用方不应该看到 `#[cfg]`）。
5. 使用 RAII 管理 `HKEY`（复用或仿照 `integration.rs` 里已有的 `RegKey`，不要手写裸 `RegCloseKey`）。
6. 为 `TrayMenu` 增加 submenu 或 icon，并补充渲染测试（新变体加进 `MenuItem` 枚举后，`to_html()` 的 `match` 会因为少一条分支而编译不过——这是编译器帮你确认渲染逻辑没漏掉新变体，不需要额外的 `_ => {}` 兜底）。

## PR 说明必须包含

- 为什么 macOS / Linux 构建不会被 Windows API 破坏；
- GUI、服务、注册表代码如何通过 `cfg` 和平台无关接口隔离；
- Win32 handle、字符串和 HRESULT 的 soundness 边界。

## 验证命令

```bash
cargo test -p v2-win-gui
cargo clippy -p v2-win-gui --all-targets -- -D warnings
cargo fmt -p v2-win-gui --check
```

如果有 Windows 机器可用，按教材第 12 课「§5.2 Windows 实机完整验证步骤」 跑一遍安装/启停服务、`reg query` 检查注册表，把关键命令输出附进 PR 描述。

## 自检清单

- [ ] 非 Windows 平台能通过平台无关测试。
- [ ] 服务状态机不依赖真实 SCM 才能测试；新增的 Pause/Continue 路径也有测试覆盖。
- [ ] 菜单渲染变化有测试覆盖；新增 `MenuItem` 变体后 `to_html()` 的 `match` 仍然穷尽。
- [ ] 注册表读取函数在非 Windows 平台有 stub，签名对调用方一致，`HKEY` 用 RAII 管理。
