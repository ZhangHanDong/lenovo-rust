# 第 5 课作业 · 性能优化与基准结论

## 作业目标

用可复现基准说明优化是否成立，而不是只凭直觉判断性能。

## 代码位置

| 用途 | 路径 |
|---|---|
| 性能与基准测试 | `projects/v2-perf-bench` |

## 需求

1. 新增第三种词频实现：`word_count_smallvec` 或“复用缓冲”版本。
2. 增加 benchmark，对比三种词频实现。
3. 对 `sum_squares` 分别测试顺序版和并行版在 `1e2`、`1e4`、`1e6` 输入规模下的表现。
4. 报告并行版本的 break-even 区间。
5. 增加至少 1 个测试，证明新增实现与已有实现结果一致。

## PR 说明必须包含

- criterion 的关键结果（贴出 `naive` / `fast` / 你新增版本三者的时间区间，以及 `sum_squares` 在 1e2/1e4/1e6 三个规模下顺序 vs 并行的时间）；
- 新实现减少了哪类成本：分配、拷贝、锁竞争或调度——**指向具体机制**，不要只写"更快了"（例如："用 `SmallVec<[u8;16]>` 替代堆分配的 `String` 累加器，减少了 N 次堆分配"，而不是"性能更好"）；
- 为什么小输入上并行版本可能更慢：说明线程调度/切分/归约的固定开销在小规模下如何超过并行节省的计算时间；
- 你观察到的 break-even 区间在哪个规模附近（例如"1e4 与 1e6 之间"），以及判断依据是什么。

## 验证命令

```bash
cargo test -p v2-perf-bench
cargo bench -p v2-perf-bench --no-run
cargo clippy -p v2-perf-bench --all-targets -- -D warnings
cargo fmt -p v2-perf-bench --check
```

完整基准可在本机运行：

```bash
cargo bench -p v2-perf-bench
```

## 自检清单

- [ ] benchmark 比较的是等价输入和等价输出。
- [ ] PR 中有具体数值，不只写“更快”。
- [ ] 性能结论没有脱离输入规模。
