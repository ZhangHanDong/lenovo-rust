# 第 15 课作业 · 综合实战交付

## 作业目标

完成一个可审查的端到端交付：规格、核心设计、平台 adapter、测试、CI、性能基线和调试报告。

## 代码位置

| 用途 | 路径 |
|---|---|
| 综合项目 | `projects/v2-capstone` |

## 需求

1. 为 `SystemReport` 增加一个有实际意义的字段，例如 `arch: CpuArch`。
2. 扩展 `HostProbe` 和 canonical string。
3. 增加至少 2 个使用 `MockProbe` 的桌面测试，证明 fingerprint 稳定且可比较。
4. 在 `#[cfg(windows)]` 下实现 Windows adapter 采集代码。
5. 非 Windows 平台保留 fallback stub。
6. 在“全新实现”或“绞杀式迁移”中选择一种路线，并在 PR 中说明。
7. 提供交付证据：交叉编译输出、三平台 CI workflow、criterion baseline、调试 AAR。

## PR 说明必须包含

- 项目规格和验收标准；
- 平台代码如何从 core 移到 adapter；
- 哪些测试证明迁移前后行为保持一致；
- Windows API、FFI 或打包链路中的主要风险。

## 验证命令

```bash
cargo build -p v2-capstone
cargo test -p v2-capstone
cargo clippy -p v2-capstone --all-targets -- -D warnings
cargo fmt -p v2-capstone --check
cargo xwin build --release -p v2-capstone --target x86_64-pc-windows-msvc
```

Windows runner 中如使用 MSI，可继续运行 cargo-wix 相关命令。

## 自检清单

- [ ] `SystemReport` 新字段进入 canonical string。
- [ ] fingerprint 对同一输入稳定。
- [ ] 平台 adapter 不污染核心模型。
- [ ] PR 中包含性能、CI 和调试证据。
