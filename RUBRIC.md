# 附录 B · 评估维度与"作业—报告"机制

## 作业机制

第 1–15 课每课一道评分作业，经 **Git PR 提交**（贴近真实评审流程，并支撑"代码审查"训练）。每道作业按下述统一维度打分（每维 **0–3**：未涉及 / 初步 / 熟练 / 精通），未触及的维度记"未涉及"。

## 七个能力维度（D1–D7）

| 能力维度 | 评分要点 | 主要相关课次 |
|---|---|---|
| **D1 正确性** | 功能满足规格、测试通过 | 全部 |
| **D2 所有权与借用** | 无不必要 clone、借用与生命周期建模合理、零拷贝权衡 | 第 1、2 课 |
| **D3 类型与错误处理** | 以类型表达状态、错误分层、`Option` / `Result` / `?` 得当 | 第 1、2 课 |
| **D4 并发、异步与性能** | 无数据竞争、`.await` 点与取消 / 背压处理正确、基准与剖析得当 | 第 3、4、5 课 |
| **D5 unsafe / FFI 边界** | `unsafe` 契约清晰、为 C/C++ 构建安全封装、跨界类型正确 | 第 6、7、15 课 |
| **D6 跨平台 / Windows / 互操作** | `windows` crate 集成、COM/WinRT、跨平台 crate 结构、cargo-xwin/MSI、桌面 GUI/系统集成、flutter_rust_bridge/diplomat 绑定正确 | 第 8–13、15 课 |
| **D7 可审查性 / 工程质量** | clippy 干净、可读、符合规格、提交规范 | 全部 |

## 评分锚点（每维 0–3 的判据）

| 分值 | 含义 | 判据示例（以 D2 为例） |
|---|---|---|
| 0 | 未涉及 | 作业未触及所有权设计 |
| 1 | 初步 | 能通过编译，但存在不必要 clone / 过度 `Arc` |
| 2 | 熟练 | 所有权建模合理，借用清晰，无冗余拷贝 |
| 3 | 精通 | 在零拷贝与可维护性间做出有理据的权衡 |

## 报告生成机制

- **个人学习评估报告（每位学员）** = 七个维度的当前水平 + 跨课次趋势 + 作业完成 / 通过率 + 结业考核得分 + 强弱项与下一步学习建议。
- **团队能力报告（整班）** = 各维度得分分布 + 团队最薄弱维度（能力缺口）+ 对企业优先领域（**Windows 跨平台 / C/C++ 迁移 / 并发异步性能**）的就绪度评估与后续发展建议。


> 因评分维度统一、可机器辅助汇总，覆盖 50–100 人时边际成本可控。
